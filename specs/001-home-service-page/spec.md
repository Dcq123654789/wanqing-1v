# Feature Specification: 上门服务页面

**Feature Branch**: `001-home-service-page`
**Created**: 2026-01-18
**Status**: Draft
**Input**: User description: "开发上门服务页面，类似商品页面，包含类型分类、两列布局，显示价格、海报、名称、销量"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 浏览和筛选上门服务 (Priority: P1)

老年人用户通过乐享生活入口进入上门服务页面，可以浏览各种类型的上门服务，并通过分类标签快速找到所需服务类型。

**Why this priority**: 这是核心功能，用户必须能够浏览和筛选服务，否则无法找到需要的服务。

**Independent Test**: 可以独立测试用户能否进入页面、查看服务列表、切换分类筛选服务，并验证筛选结果正确显示对应分类的服务。

**Acceptance Scenarios**:

1. **Given** 用户进入上门服务页面，**When** 页面加载完成，**Then** 显示所有可用的上门服务列表
2. **Given** 用户在服务列表页面，**When** 点击某个分类标签（如"家政清洁"），**Then** 列表只显示该分类下的服务
3. **Given** 用户选择了某个分类，**When** 点击"全部"分类，**Then** 列表恢复显示所有服务
4. **Given** 服务列表有数据，**When** 用户浏览列表，**Then** 每个服务卡片显示海报图片、服务名称、价格和销量信息

---

### User Story 2 - 查看服务详情 (Priority: P2)

用户对某个上门服务感兴趣时，点击服务卡片可以查看服务的详细信息，包括服务描述、服务内容、价格说明等。

**Why this priority**: 查看详情是用户决策的关键步骤，帮助用户了解服务的具体内容后再决定是否预约。

**Independent Test**: 可以独立测试用户能否从列表页进入详情页，以及详情页是否正确展示服务的完整信息。

**Acceptance Scenarios**:

1. **Given** 用户在服务列表页，**When** 点击某个服务卡片，**Then** 跳转到该服务的详情页面
2. **Given** 用户进入服务详情页，**When** 页面加载完成，**Then** 显示服务的详细信息（海报、名称、价格、销量、描述、规格等）

---

### User Story 3 - 价格排序功能 (Priority: P3)

用户可以按照价格高低对服务进行排序，方便找到符合预算的服务。

**Why this priority**: 排序功能是增强体验的辅助功能，帮助用户更快做出决策，但不影响核心浏览流程。

**Independent Test**: 可以独立测试用户能否点击排序按钮，以及服务列表是否按照选择的价格顺序正确排列。

**Acceptance Scenarios**:

1. **Given** 用户在服务列表页，**When** 点击"价格排序"按钮一次，**Then** 服务按价格从低到高排列
2. **Given** 服务已按价格从低到高排序，**When** 再次点击"价格排序"按钮，**Then** 服务按价格从高到低排列
3. **Given** 服务已排序，**When** 第三次点击"价格排序"按钮，**Then** 取消排序，恢复默认排列

---

### Edge Cases

- 当某个分类下没有服务时，显示"暂无服务"的空状态提示
- 当服务海报图片加载失败时，显示默认占位图片
- 当服务价格为0或免费时，显示"免费"而非价格数字
- 当销量为0时，显示"暂无销量"或合适文案
- 当分类标签过多超出屏幕宽度时，支持横向滚动查看更多分类

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供上门服务列表页面，展示所有可用的上门服务
- **FR-002**: 系统必须支持按服务类型进行分类筛选（至少包含：家政清洁、维修服务、护理照护、生活服务等分类）
- **FR-003**: 系统必须以两列网格布局展示服务列表，每行显示两个服务卡片
- **FR-004**: 每个服务卡片必须显示服务海报图片（比例1:1）、服务名称、服务价格、销售数量
- **FR-005**: 系统必须支持价格排序功能，允许用户在"价格从低到高"、"价格从高到低"、"取消排序"三种状态间切换
- **FR-006**: 分类标签栏必须支持横向滚动，以容纳更多分类选项
- **FR-007**: 系统必须在服务数据加载时显示加载状态，避免用户等待时的困惑
- **FR-008**: 系统必须在筛选结果为空时显示友好的空状态提示
- **FR-009**: 系统必须支持点击服务卡片跳转到服务详情页，传递服务ID参数
- **FR-010**: 系统必须确保服务海报图片采用懒加载优化，提升页面加载性能

### Key Entities

- **上门服务（HomeService）**: 代表一个可预约的上门服务项目，包含服务ID、名称、海报图片、价格、销量、服务分类、服务描述等属性
- **服务分类（ServiceCategory）**: 代表服务的类型分类，包含分类ID、分类名称、分类图标等属性
- **服务详情（ServiceDetail）**: 服务的完整信息，在服务实体的基础上扩展描述、多图展示、规格说明等详细信息

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可以在3秒内完成服务分类切换并看到筛选结果
- **SC-002**: 页面支持至少50个服务项目的流畅滚动浏览，无明显卡顿
- **SC-003**: 用户能够通过分类筛选在10秒内找到目标类型的服务
- **SC-004**: 服务卡片信息完整度达到100%，所有卡片均正确显示海报、名称、价格、销量
- **SC-005**: 图片加载失败时显示占位图的比例为100%，避免页面布局错乱

## Assumptions

1. 上门服务数据通过模拟数据（mock data）提供，暂未对接后端API
2. 服务海报图片使用占位图片服务（如 picsum.photos）进行演示
3. 服务详情页面功能将在后续迭代中完善
4. 分类标签的数量和名称可以根据业务需求灵活调整
5. 用户群体主要为老年人，因此界面设计需要考虑大字体、高对比度等无障碍特性
